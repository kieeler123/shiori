🇯🇵 日本語（Dev Log）
📌 問題の概要

profiles.nickname を shiori_items の一覧画面で表示したかったが、
常に「匿名」と表示されてしまった。

Supabase の join を使って

profile:profiles(nickname)


で取得していたが、画面では nickname が表示されなかった。

🔎 原因分析
① DB / RLS の問題ではなかった

console.log で確認すると：

profile: { nickname: "장준" }


つまり、DB からは正しくデータが取得できていた。

② TypeScript の型が実際のレスポンスと一致していなかった

実際のレスポンス：

profile: { nickname: string } | null


しかし型定義では：

profiles と書いていた

あるいは { nickname }[] の配列と想定していた

そのため：

as DbLogRow[] で型エラー発生

UI 側で undefined 扱いになり「匿名」表示

③ insert / update 時に join を同時に取得しようとしていた

TABLE_BASE（元テーブル）で insert / update 後に
profile:profiles(nickname) を含めて .select() していた。

View + Join の関係で型や join の挙動が不安定になる可能性があった。

✅ 解決方法
1️⃣ alias を統一
profile:profiles(nickname)


結果フィールドを profile に固定。

2️⃣ DbLogRow の型を実際のレスポンスに合わせる
profile: { nickname: string | null } | null;


配列ではなくオブジェクトとして定義。

3️⃣ TypeScript キャストの修正
return (data ?? []) as unknown as DbLogRow[];


構造の不一致による型エラーを回避。

4️⃣ insert / update 後は View から再取得

書き込み → 元テーブル

表示用取得 → View

という責務分離を徹底。

🧠 学んだこと

Supabase の join は DB カラムではなく select が作るフィールド

alias の有無でレスポンス形状が変わる

View + Join + TypeScript は必ず実レスポンス shape を確認する

insert/update 直後の join 取得は不安定になる可能性あり

🇺🇸 English (Dev Log)
📌 Issue Summary

I tried to display profiles.nickname in the shiori_items list view,
but it always showed "Anonymous".

The join query was:

profile:profiles(nickname)


However, nickname was not rendered correctly.

🔎 Root Cause
1) Database and RLS were not the issue

Console log confirmed:

profile: { nickname: "장준" }


So the join worked correctly.

2) TypeScript type mismatch

Actual response shape:

profile: { nickname: string } | null


But the type definition assumed:

profiles instead of profile

or an array { nickname }[]

This caused:

Type assertion errors

UI resolving nickname as undefined

"Anonymous" fallback being triggered

3) insert/update tried to fetch join result directly

Using join select immediately after insert/update on base table
caused potential inconsistency with view + relationship resolution.

✅ Solution
1) Unified alias usage
profile:profiles(nickname)

2) Adjusted DbLogRow type
profile: { nickname: string | null } | null;

3) Fixed type casting
return (data ?? []) as unknown as DbLogRow[];

4) Separated write and read responsibility

Write → base table

Read → view (with join)

🧠 Lessons Learned

Supabase join fields are virtual (generated by select)

Alias determines response shape

Always inspect real response before defining types

View + Join + TypeScript requires careful alignment

Re-fetching from view after insert/update is safer

🇰🇷 한국어 (Dev Log)
📌 문제 요약

shiori_items 목록에서 profiles.nickname을 표시하려 했으나
항상 "익명"으로 표시되었다.

join은 다음과 같이 사용했다:

profile:profiles(nickname)


하지만 화면에 닉네임이 나타나지 않았다.

🔎 원인 분석
1) DB나 RLS 문제는 아니었다

console.log 확인 결과:

profile: { nickname: "장준" }


DB에서는 정상적으로 내려오고 있었다.

2) TypeScript 타입이 실제 응답 구조와 달랐다

실제 응답:

profile: { nickname: string } | null


하지만 타입에서는:

profiles로 정의하거나

배열 형태 { nickname }[]로 가정하고 있었다

그 결과:

타입 에러 발생

UI에서 undefined 처리

결국 "익명" fallback 실행

3) insert/update 직후 join을 바로 select하려 했던 구조

원본 테이블에서 write 후
join 포함 select를 바로 시도하면서 구조가 불안정해질 가능성 존재

✅ 해결 방법

alias를 profile로 통일

DbLogRow 타입을 실제 응답 구조에 맞게 수정

타입 캐스팅은 unknown을 한 번 거쳐 안정화

write와 read를 분리 (write → base, read → view)

🧠 이번에 얻은 교훈

Supabase join 필드는 실제 DB 컬럼이 아니라 select가 생성하는 가상 필드

alias에 따라 응답 구조가 달라진다

타입 정의 전에 반드시 실제 응답 shape 확인

View + Join + TS는 구조 정합성이 매우 중요

insert/update 후 view 재조회 방식이 가장 안정적

이번 건은 단순 버그가 아니라
“Supabase 관계 + View 설계 + 타입 안정성”을 한 단계 이해한 사건이야.
