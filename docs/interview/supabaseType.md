🇯🇵 日本語（面接 Q&A）
❓ Q1. Supabase の join で発生した問題について教えてください。

A.
shiori_items の一覧画面で投稿者の nickname を表示しようとしましたが、
常に「匿名」と表示されてしまいました。

join は以下のように書いていました：

profile:profiles(nickname)


最初は RLS やデータ欠損を疑いましたが、console.log で確認すると
profile: { nickname: "장준" } が正常に取得できていました。

❓ Q2. では原因は何だったのですか？

A.
原因は TypeScript の型定義が実際のレスポンス構造と一致していなかったことです。

実際のレスポンスは：

profile: { nickname: string } | null


しかし、型では：

profiles と書いていた

または { nickname }[] の配列と想定していた

その結果、UI 側で undefined となり、「匿名」が表示されていました。

❓ Q3. どのように解決しましたか？

A.

alias を profile に統一

DbLogRow の型を実際のレスポンスに合わせて修正

insert/update 後は View から再取得する設計に変更

型キャストを unknown を経由する形で安全に処理

❓ Q4. この経験から何を学びましたか？

A.

Supabase の join フィールドは実際の DB カラムではなく、select が生成する仮想フィールドである

alias によってレスポンス形状が変わる

View + Join + TypeScript の組み合わせでは、実際のレスポンスを必ず確認するべき

write と read の責務分離は設計上非常に重要

🇺🇸 English (Interview Q&A)
❓ Q1. Can you describe a technical issue you faced with Supabase joins?

A.
I tried to display the author's nickname in a list view using:

profile:profiles(nickname)


However, the UI kept showing "Anonymous."

At first, I suspected RLS or missing profile rows, but console logs confirmed that the response correctly included:

profile: { nickname: "장준" }

❓ Q2. What was the root cause?

A.
The issue was a mismatch between the TypeScript type definition and the actual API response shape.

The real response was:

profile: { nickname: string } | null


But my type definition assumed:

profiles instead of profile

or an array { nickname }[]

This caused undefined values in the UI and triggered the fallback.

❓ Q3. How did you resolve it?

A.

Unified the alias to profile

Updated the DbLogRow type to match the actual response

Separated write and read responsibilities (write to base table, read from view)

Used unknown before casting to stabilize TypeScript compatibility

❓ Q4. What did you learn from this experience?

A.

Supabase join fields are virtual fields generated by select

Alias determines response shape

Always inspect the real API response before finalizing type definitions

View + Join + TypeScript requires careful structural alignment

Separating write and read logic improves stability

🇰🇷 한국어 (면접 Q&A)
❓ Q1. Supabase join 관련해서 겪었던 문제를 설명해 주세요.

A.
shiori_items 목록 화면에서 작성자의 nickname을 표시하려 했지만
항상 "익명"으로 표시되는 문제가 있었습니다.

join은 다음과 같이 사용했습니다:

profile:profiles(nickname)


처음에는 RLS 문제나 데이터 누락을 의심했지만,
console.log로 확인해보니 실제로는 nickname이 정상적으로 내려오고 있었습니다.

❓ Q2. 그럼 실제 원인은 무엇이었나요?

A.
TypeScript 타입 정의가 실제 응답 구조와 일치하지 않았던 것이 원인이었습니다.

실제 응답 구조는:

profile: { nickname: string } | null


하지만 타입에서는:

profiles로 정의하거나

배열 형태 { nickname }[]로 가정하고 있었습니다.

그 결과 UI에서 undefined로 처리되어 "익명" fallback이 실행되었습니다.

❓ Q3. 어떻게 해결했나요?

A.

alias를 profile로 통일

DbLogRow 타입을 실제 응답 구조에 맞게 수정

write와 read를 분리 (write → base table, read → view)

타입 캐스팅을 unknown을 거쳐 안정화

❓ Q4. 이 경험을 통해 무엇을 배웠나요?

A.

Supabase join 필드는 실제 DB 컬럼이 아니라 select가 생성하는 가상 필드라는 점

alias에 따라 응답 구조가 달라진다는 점

타입 정의 전에 반드시 실제 응답 구조를 확인해야 한다는 점

View + Join + TypeScript 조합에서는 구조 정합성이 매우 중요하다는 점

write와 read를 분리하면 시스템 안정성이 높아진다는 점
