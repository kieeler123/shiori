🇯🇵 面接想定 Q&A（今日の経験ベース）
❓ Q1. 最近直面した技術的な問題と、その解決過程を教えてください。

A.
Shioriという個人プロジェクトにログイン機能を追加した際、Googleログインが正常に動作しない問題が発生しました。
エラーメッセージは「このアプリのリクエストは無効です」でした。

最初はフロントエンドやSupabase設定の問題だと考えましたが、原因はGoogle Cloud Consoleで作成したOAuthクライアントの種類ミスでした。

「Web application」タイプではないClient IDでは、正しいRedirect URIが設定できず、認証フロー自体が成立しませんでした。

この経験から、認証処理はコードだけでなく、外部サービスとの構造的な接続理解が重要だと学びました。

❓ Q2. 認証（Authentication）と認可（Authorization）の違いをどう理解していますか？

A.
今回Supabaseでログイン後にも401エラーが発生しました。
これは認証の問題ではなく、RLS（Row Level Security）による認可制御でした。

認証：ユーザーが誰かを証明する

認可：そのユーザーが何をできるかを制御する

つまりログイン成功＝データ操作可能ではないことを実体験で理解しました。
DBは「接続できるか」ではなく「その行に触れる権限があるか」で動いています。

❓ Q3. フロントエンドのバグだと思ったが、実は違ったケースはありますか？

A.
はい。セッションが取得できているのにUIが切り替わらない問題がありました。

最初は認証失敗だと思いましたが、原因はReact側の状態管理条件でした。
sessionの有無だけでなく、ready状態やレンダリング条件の順序が重要でした。

この経験で、問題が発生したときに「どの層の問題か」を切り分ける習慣がつきました。

❓ Q4. この経験から何を学びましたか？

A.
エラーはスキル不足ではなく「理解の範囲外」から来ることを学びました。

しかし構造を一つずつ理解していくと、システム全体の見え方が変わりました。
今回の経験で、フロントエンド開発から「運用視点の開発」に一歩進んだと感じています。

🇰🇷 면접용 Q&A
❓ Q1. 최근 겪었던 기술적인 문제와 해결 과정을 설명해 주세요.

A.
개인 프로젝트 Shiori에 로그인 기능을 추가하던 중 Google 로그인이 동작하지 않는 문제가 발생했습니다.
에러는 “이 앱의 요청이 유효하지 않습니다”였습니다.

프론트 코드와 Supabase 설정은 정상이었고, 원인은 Google Cloud Console에서 OAuth Client 타입을 잘못 만든 것이었습니다.

Web application 타입이 아니면 Redirect URI가 존재하지 않아 인증 흐름이 성립하지 않습니다.

이 경험을 통해 인증 처리는 코드 문제가 아니라, 외부 시스템과의 구조 이해가 핵심이라는 것을 배웠습니다.

❓ Q2. 인증(Authentication)과 인가(Authorization)의 차이를 어떻게 이해하고 있나요?

A.
로그인 성공 후에도 Supabase에서 401 에러가 발생했습니다.
원인은 인증이 아니라 RLS(Row Level Security)였습니다.

인증: 사용자가 누구인지 확인

인가: 그 사용자가 무엇을 할 수 있는지 결정

로그인했다고 해서 DB 작업이 가능한 것이 아니라,
DB는 행 단위 권한을 기준으로 동작한다는 것을 실제로 경험했습니다.

❓ Q3. 프론트엔드 문제라고 생각했지만 다른 원인이었던 사례가 있나요?

A.
세션이 로드됐는데 UI가 바뀌지 않는 문제가 있었습니다.
처음엔 인증 실패로 생각했지만, 실제 원인은 React 상태 조건 처리 문제였습니다.

이 경험 이후로 문제 발생 시 “어느 레이어의 문제인지” 먼저 분리해서 생각하게 되었습니다.

❓ Q4. 이 경험을 통해 무엇을 배웠나요?

A.
에러는 실력 부족이 아니라 이해 범위 밖에서 발생한다는 걸 배웠습니다.
하지만 구조를 하나씩 이해하면 시스템 전체가 보이기 시작합니다.

이번 경험으로 단순 기능 구현 수준을 넘어
운영 구조를 고려하는 개발 단계로 올라섰다고 느꼈습니다.
